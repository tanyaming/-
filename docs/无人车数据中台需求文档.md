# 无人车数据中台需求文档

版本：V0.1  
日期：2026-05-28  
阶段：需求梳理稿，待确认后进入设计与开发

## 1. 背景与目标

本项目需要建设一个无人车数据中台，用于接入不同无人车厂商的数据，并按不同城市监管平台的协议要求进行转换、校验、补齐和上报。首期范围包括：

- 监管平台：成都市智能网联汽车监管平台。
- 无人车厂商：新石器、九识。
- 技术选型：后端 Python FastAPI，前端 Vue，数据库 MySQL。

中台不能只为当前三份协议做一次性对接，需要具备后续扩展能力：

- 支持新增更多无人车厂商的数据接入。
- 支持新增更多城市监管平台的数据上报。
- 支持同一车辆或同一厂商按城市、项目、车辆分组配置不同监管上报规则。
- 支持协议字段、枚举、频率、坐标系、证书、鉴权方式的配置化和插件化扩展。

## 2. 文档依据

本需求基于以下接口文档整理：

- 《车辆接入监管平台对接协议V1.0.4》
- 《新石器无人车接口文档（2025.07）》
- 《九识通用推送文档V2.12》

## 3. 总体原则

1. 标准模型优先：厂商数据先转换为中台统一车辆数据模型，再由城市监管适配器转换为目标平台协议。
2. 双向适配解耦：厂商接入适配器和城市监管适配器互不直接依赖。
3. 配置驱动扩展：鉴权参数、车辆映射、城市平台地址、TLS 证书、上报频率、字段映射和枚举映射应通过后台配置管理。
4. 原始数据可追溯：保存厂商原始报文、标准化数据、监管上报报文和上报结果，便于排障和审计。
5. 实时链路独立：高频车辆状态上报、低频基础资料同步、业务状态同步、故障同步应拆分处理，避免互相阻塞。
6. 明确数据质量：对于厂商无法提供的监管必填字段，必须标记数据来源、默认值策略和异常值策略，不能静默伪造。

## 4. 首期协议摘要

### 4.1 成都市监管平台

成都市平台采用车端到平台的 TCP over TLS 长连接协议，不是 HTTP 接口。

核心要求：

- 平台地址：`tcp://182.148.54.57:38090`
- 认证方式：TLS x509 证书双向认证。
- 连接形式：原则上每辆车一条独立 TCP/TLS 连接。
- 会话保持：平台连续 5 分钟未收到车辆上报消息会断开连接。
- 数据包：固定 16 字节报头 + 可变数据段，二进制编码。
- 坐标要求：协议说明中写明 WGS84，同时准静态参数新增坐标系类型字段，需支持 GCJ-02/WGS-84 可配置。
- 准静态参数消息 `0x34`：连接认证成功后首次发送，包含车辆编号、软硬件版本、无线通讯类型、定位精度、时间同步方式、坐标系类型。
- 运行状态消息 `0x15`：车辆运行状态中上报频率 10Hz。
- 故障消息 `0x5c`：有故障时立即发送并按 1Hz 上报，故障消除后可降低至 1 分钟一次。
- 平台响应准静态参数 `0x35`：非 0 状态会立即断开连接。

成都市车辆编号为固定 8 位 UTF-8，格式为 `AABCDDDD`，需在平台登记并与车辆证书匹配。

### 4.2 新石器接口

新石器接口为 HTTP API，由中台主动拉取或调用。

核心要求：

- 测试环境：`https://scapi.test.neolix.net/`
- 线上环境：`https://scapi.neolix.net/`
- 请求头：`X-From` 唯一标识、`X-Version` 版本号。
- Token：通过 `/auth/oauth/token` 获取 `access_token`，授权方式为 `client_credentials`。
- 签名：`client_secret`、秒级时间戳、两位随机数按字典序排序后拼接，使用 SHA-1。
- 坐标：接口响应坐标为 GCJ-02。
- 车辆列表：`GET /openapi-server/slvapi/getVehicleList`
- 单车状态：`POST /openapi-server/slvapi/getVehicleInfo`
- 批量实时基础信息：`POST /openapi-server/slvapi/batchGetVehicleList`
- 车辆基础信息 T+1：`POST /openapi-server/slvapi/getVehicleBaseList`
- 其他接口：站点、任务、路线、取消任务、货箱、上下电、开关笼、音量等。

首期监管上报优先使用车辆列表、车辆基础信息、批量实时基础信息、单车状态接口。任务控制类接口作为二期或后台运维能力预留。

### 4.3 九识接口

九识接口为 MQTT 推送，由中台作为 MQTT 客户端订阅。

核心要求：

- 企业需要提供给九识 MQTT server 的 host、用户名、密码。
- 九识提供 topic 中的 `organizationCode`。
- 实时数据 topic：`{organizationCode}/vehicle/+/realtime/push`
- 实时数据推送条件：车辆开机且正常联网。
- 实时数据推送频率：1Hz。
- 业务状态 topic：`{organizationCode}/vehicle/+/business/push`
- 业务状态推送条件：业务状态变化时推送一次。
- 坐标：实时数据为 GCJ-02。
- 实时字段包括车辆号、车牌、VIN、经纬度、朝向、速度、电量、电池温度、里程、任务里程、车辆模式、车辆状态、配送状态、充电状态、油门、刹车、灯光、档位、方向盘角、胎压、货箱门锁、冷机温度、货箱温度等。
- 业务字段包括配送状态、取消信息、当前停靠点、上一个停靠点、规划里程、预计耗时、订单列表等。

## 5. 关键差异与约束

| 项目 | 成都市监管平台 | 新石器 | 九识 |
| --- | --- | --- | --- |
| 协议类型 | TCP over TLS 二进制长连接 | HTTP API | MQTT 推送 |
| 数据方向 | 中台模拟车端上报监管平台 | 中台主动拉取 | 厂商主动推送到中台订阅端 |
| 鉴权 | 每车 TLS x509 证书 | OAuth Token + SHA-1 签名 | MQTT host/用户名/密码 |
| 坐标 | WGS84 为主，协议含坐标系字段 | GCJ-02 | GCJ-02 |
| 状态频率 | 运行状态 10Hz | 文档未明确可支持 10Hz | 实时数据 1Hz |
| 故障频率 | 故障 1Hz，无故障 1分钟 | 批量接口含 faultStatus | vehicleState/MALFUNCTION 等 |

重要风险：

- 九识文档实时数据仅 1Hz，无法天然满足成都市监管平台 10Hz 的真实采样频率。
- 新石器 HTTP 接口未明确高频拉取频率、限流策略和数据刷新频率，是否能支撑 10Hz 需向厂商确认。
- 若源数据低于监管要求，中台只能采用保活、最近值复用或插值策略，但这些数据不应被标记为真实 10Hz 采样。
- 成都市平台要求车辆编号与证书对应，证书管理和车辆映射必须严格校验。

## 6. 业务范围

### 6.1 首期范围

1. 厂商接入
   - 新石器 HTTP 账号配置、Token 管理、签名、车辆列表同步、车辆基础信息同步、车辆实时状态拉取。
   - 九识 MQTT 连接配置、订阅实时数据、订阅业务状态、断线重连、消息解析。

2. 标准化处理
   - 将新石器、九识车辆数据转换为中台统一车辆状态模型。
   - 支持 GCJ-02 到 WGS84 坐标转换。
   - 支持速度、角度、里程、胎压、电量、温度、档位、驾驶模式、故障状态等单位和枚举转换。
   - 支持必填字段校验、缺省值策略、异常值策略。

3. 成都市监管上报
   - 按车辆建立 TCP/TLS 连接。
   - 连接成功后发送准静态参数 `0x34`。
   - 按配置周期发送运行状态 `0x15`。
   - 按故障状态发送故障消息 `0x5c`。
   - 处理平台准静态参数响应 `0x35`。
   - 支持连接断开后自动重连。

4. 管理后台
   - 厂商账号配置。
   - 城市监管平台配置。
   - 车辆档案与厂商车辆映射。
   - 车辆与城市监管平台绑定。
   - 证书上传和绑定。
   - 实时状态查看。
   - 上报链路状态查看。
   - 异常和日志查看。

5. 数据留存
   - 厂商原始消息。
   - 标准化车辆状态快照。
   - 监管上报记录。
   - 车辆档案、厂商配置、监管平台配置、证书元数据、字段映射配置。

### 6.2 暂不纳入首期

- 通过中台下发车辆控制指令。
- 任务调度、路线规划、订单运营系统。
- 监管平台之外的业务大屏。
- 多租户计费和复杂权限模型。

上述能力可作为二期扩展预留接口和数据模型。

## 7. 系统角色

- 系统管理员：维护平台全局配置、用户、角色、字典。
- 运维人员：维护厂商账号、MQTT 连接、监管平台连接、证书、车辆映射，处理异常。
- 业务查看人员：查看车辆实时状态、上报状态、统计数据。
- 开发/集成人员：维护字段映射、枚举映射、适配器插件。

## 8. 总体架构需求

建议采用分层和适配器架构：

```text
厂商接口层
  - Neolix HTTP Adapter
  - Jiushi MQTT Adapter
  - Future Vendor Adapter

标准化处理层
  - 原始数据解析
  - 字段映射
  - 枚举转换
  - 坐标转换
  - 单位转换
  - 数据质量校验

中台核心层
  - 车辆档案
  - 车辆实时状态
  - 车辆故障状态
  - 业务状态
  - 上报任务调度
  - 事件与日志

城市监管适配层
  - Chengdu TCP/TLS Binary Adapter
  - Future City Adapter

管理后台
  - Vue Admin
  - FastAPI Admin API

基础设施
  - MySQL
  - Redis/队列，建议用于实时缓存、任务调度、限流和消息缓冲
  - 日志与监控
```

说明：用户指定数据库为 MySQL。考虑到车辆状态高频处理和连接调度，建议增加 Redis 或消息队列作为运行时组件。如果首期需要极简部署，可先用 MySQL + 进程内队列实现，但并发车辆数增加后需要升级。

## 9. 核心功能需求

### 9.1 厂商管理

- 新增、编辑、启停厂商。
- 配置厂商类型：新石器、九识、其他。
- 配置环境：测试、生产。
- 配置鉴权信息：
  - 新石器：base_url、client_id、client_secret、userId、X-From、X-Version。
  - 九识：mqtt_host、port、username、password、organizationCode、topic、QoS。
- 敏感字段加密存储。
- 支持连接测试。
- 支持厂商同步状态查看。

### 9.2 车辆档案管理

- 维护中台车辆主档，包括 VIN、厂商车辆 ID、厂商车辆号、车牌、车型、动力类型、车辆类别、所属厂商、所属项目、所属城市。
- 维护成都市监管车辆编号 `AABCDDDD`。
- 校验监管车辆编号长度和格式。
- 支持车辆启用、停用。
- 支持车辆与监管平台、TLS 证书绑定。
- 支持批量导入车辆。
- 支持从新石器、九识接口自动发现车辆并进入待绑定列表。

### 9.3 新石器接入

- 获取并缓存 access_token，过期前自动刷新。
- 对需要签名的请求自动生成 signature、timeStamp、nonce。
- 定时同步车辆列表。
- 定时同步车辆基础信息 T+1。
- 按配置周期拉取批量车辆实时信息。
- 对单车补充拉取状态信息。
- 处理 HTTP 超时、限流、Token 失效、签名失败、业务错误码。
- 保存原始响应和解析结果。

### 9.4 九识接入

- 建立 MQTT 连接并订阅实时和业务 topic。
- 支持断线重连、订阅恢复、消息去重。
- 解析实时车辆数据。
- 解析业务状态数据。
- 根据 topic 或 payload 识别车辆。
- 保存原始消息和解析结果。
- 处理异常 JSON、字段缺失、时间戳异常、车辆未绑定等问题。

### 9.5 标准车辆状态模型

中台内部需要定义统一模型，至少包括：

- vehicle_id：中台车辆 ID。
- vendor_code：厂商编码。
- vendor_vehicle_id：厂商车辆标识。
- vin：车架号。
- regulatory_vehicle_no：监管车辆编号。
- source_timestamp：厂商采集或上报时间。
- received_at：中台接收时间。
- location：经度、纬度、高程、坐标系。
- motion：速度、航向角、纵向/横向/垂向加速度、横摆角速度。
- control：档位、方向盘转角、油门、制动踏板、驾驶模式。
- energy：电量、电池温度、续航、充电状态、电压、电流。
- body：车灯、车门、货箱门、锁状态、喇叭。
- tires：车轮数、轮速、胎压。
- adas：ABS、TCS、ESP、LKA、ACC、FCW、LDW、AEB、LCA、DMS。
- fault：车辆故障、硬件故障、软件故障。
- business：任务状态、配送状态、站点、订单摘要。
- quality：字段完整度、异常字段、是否复用最近值、是否插值。

### 9.6 字段转换与映射

必须支持以下转换能力：

- 坐标转换：GCJ-02 转 WGS84，按城市监管平台配置决定输出坐标系。
- 速度转换：
  - 九识 m/s 到成都市 0.01 m/s。
  - 新石器 km/h 到 m/s 后再转成成都市传输值。
- 航向转换：
  - 九识 heading 为弧度且正东为 0，需转换为成都市正北顺时针角度。
  - 新石器 gnssHead 按文档值确认单位后转换。
- 里程转换：
  - 九识 odometer 米，成都市里程单位 0.1 km。
  - 新石器 mile 续航里程不等于总里程，需区分使用。
- 档位映射：
  - 九识 `GEAR_DRIVE/REVERSE/PARKING/NEUTRAL` 映射到成都市 D/R/P/N。
  - 新石器 `D` 等映射到成都市枚举。
- 驾驶模式映射：
  - 新石器 driveMode 映射人工、自动、远程等监管枚举。
  - 九识 chassisState 映射监管驾驶模式。
- 灯光、门锁、胎压、故障位映射。
- 监管必填字段缺失时，按字段级策略处理：使用异常值、缺省值、最近值、禁发或告警。

### 9.7 成都市监管上报

- 为每辆已绑定车辆维护独立连接上下文。
- 支持按车辆配置证书、私钥、CA 证书。
- 建立 TLS 双向认证连接。
- 首次连接发送准静态参数。
- 准静态参数响应非确认时，记录异常并停止该车辆上报。
- 运行状态消息按配置频率调度，默认目标为 10Hz。
- 源数据不足 10Hz 时，需支持配置策略：
  - 严格模式：低于频率则告警，按真实源频率上报或停止上报。
  - 保活模式：复用最近一次状态补齐频率，并标记质量。
  - 插值模式：对位置、速度做简单插值，并标记质量。
- 故障消息按故障状态动态调度：
  - 有故障：立即发送并 1Hz。
  - 无故障或故障消除：1 分钟一次。
- 每条上报维护消息编号，自增，溢出后从 1 开始。
- 支持二进制报文编码、长度校验、字段范围校验、异常值填充。
- 记录连接状态、最近上报时间、最近错误、重连次数。

### 9.8 管理后台

后台页面建议包括：

- 登录页。
- 首页概览：车辆总数、在线数、上报中车辆数、异常车辆数、厂商连接状态、监管连接状态。
- 厂商管理：新石器、九识连接配置、状态、测试连接、同步按钮。
- 监管平台管理：城市、平台地址、协议类型、证书要求、上报频率、启停。
- 车辆管理：车辆档案、厂商映射、监管编号、证书绑定、上报状态。
- 证书管理：上传、到期时间、绑定车辆、启停、下载审计。
- 实时状态：车辆当前位置、速度、电量、驾驶模式、故障、更新时间。
- 上报监控：每车连接状态、报文类型、发送频率、成功/失败计数。
- 原始数据日志：按厂商、车辆、时间查询。
- 上报日志：按城市平台、车辆、报文类型、时间查询。
- 告警中心：连接失败、字段缺失、证书异常、上报失败、源数据超时。
- 字典与映射：厂商枚举到中台枚举，中台枚举到监管枚举。
- 用户和角色：基础 RBAC。

## 10. 数据库需求

首期建议 MySQL 表包括：

- users：用户。
- roles：角色。
- vendor_accounts：厂商账号配置。
- regulatory_platforms：城市监管平台配置。
- vehicles：中台车辆档案。
- vehicle_vendor_bindings：车辆与厂商标识映射。
- vehicle_regulatory_bindings：车辆与监管平台绑定。
- vehicle_certificates：车辆证书元数据。
- field_mapping_rules：字段映射规则。
- enum_mapping_rules：枚举映射规则。
- vehicle_latest_states：车辆最新标准状态。
- vehicle_raw_messages：厂商原始消息。
- vehicle_normalized_messages：标准化消息。
- regulatory_report_logs：监管上报日志。
- connection_status：厂商和监管连接状态。
- alert_events：告警事件。
- operation_logs：后台操作日志。

高频运行状态是否全量落 MySQL 需要确认保存周期和车辆规模。建议：

- 最新状态必须保存。
- 原始消息和上报日志可按采样、分区或冷热策略保存。
- 高频明细后续可引入时序数据库或对象存储，MySQL 保存索引和摘要。

## 11. API 需求

FastAPI 对后台提供 REST API：

- `/api/auth/*`：登录、登出、当前用户。
- `/api/vendors/*`：厂商账号配置、连接测试、启停、同步。
- `/api/regulatory-platforms/*`：城市监管平台配置。
- `/api/vehicles/*`：车辆管理、绑定、导入、启停。
- `/api/certificates/*`：证书上传、绑定、到期检查。
- `/api/states/*`：实时状态查询。
- `/api/reports/*`：监管上报状态与日志。
- `/api/raw-messages/*`：原始消息查询。
- `/api/mappings/*`：字段和枚举映射。
- `/api/alerts/*`：告警查询和处理。

内部服务接口需区分外部后台 API 和内部调度/适配器 API，避免后台直接操作底层连接。

## 12. 非功能需求

### 12.1 性能

- 支持首期不少于 100 辆车接入和状态监控。
- 成都市 10Hz 上报在 100 辆车规模下约 1000 条运行状态报文/秒，编码和调度模块需异步化。
- 后台查询不能影响实时采集和上报链路。

### 12.2 可用性

- 厂商连接失败自动重试。
- MQTT 断线自动重连。
- 成都市 TCP/TLS 连接断开自动重连。
- 单车异常不能影响其他车辆。
- 支持服务重启后恢复车辆上报任务。

### 12.3 安全

- 厂商密钥、MQTT 密码、证书私钥加密存储。
- 证书文件访问最小权限。
- 后台 API 需要鉴权和角色权限。
- 操作日志记录敏感配置变更。
- 不在日志中明文输出 access_token、client_secret、证书私钥。

### 12.4 可观测性

- 记录接口调用耗时、成功率、错误码。
- 记录 MQTT 消息接收数量、延迟、解析失败数量。
- 记录监管上报频率、连接状态、重连次数、失败原因。
- 对源数据超时、字段缺失、证书过期、上报失败生成告警。

## 13. 分期计划

### 13.1 第一期：成都市监管上报闭环

- 建立项目骨架：FastAPI、Vue、MySQL。
- 完成车辆档案、厂商配置、监管配置、证书管理。
- 完成新石器和九识首期数据接入。
- 完成统一车辆状态模型。
- 完成成都市二进制协议编码和 TLS 上报。
- 完成基础后台监控和日志查询。
- 完成端到端联调。

### 13.2 第二期：扩展和运营能力

- 支持更多厂商适配器。
- 支持更多城市监管平台适配器。
- 支持可视化字段映射配置。
- 支持车辆地图、轨迹回放和统计报表。
- 支持任务、订单、路线等业务数据接入。
- 支持控制类接口统一封装。

### 13.3 第三期：规模化与高可用

- 引入消息队列或流处理。
- 引入时序存储或冷热归档。
- 支持多实例部署和连接调度分片。
- 支持租户和项目隔离。
- 支持自动化协议测试和模拟车端。

## 14. 待确认问题

1. 成都市平台是否接受准静态参数中坐标系类型为 GCJ-02，还是必须统一转 WGS84 后上报。
2. 新石器接口的实时数据刷新频率、调用限流、批量接口最大 VIN 数量。
3. 九识是否可以提供高于 1Hz 的实时数据，或是否有车端直连/高频接口。
4. 对于源数据不足 10Hz 的车辆，监管侧是否允许最近值复用或低频上报。
5. 每辆车的成都市 TLS 证书获取、命名、存储和更新流程。
6. 首期车辆规模、预计并发连接数、数据留存周期。
7. 是否需要首期接入任务、订单、站点、路线等业务数据，还是仅做监管必需数据。
8. 是否需要支持多企业、多项目隔离。
9. 后台权限是否只需管理员/运维/查看三类角色。
10. 是否有监管平台联调环境和测试车辆编号。

## 15. 初步验收标准

- 能在后台配置新石器账号并成功同步车辆和实时状态。
- 能在后台配置九识 MQTT 并成功接收实时和业务推送。
- 能将两家厂商数据转换为统一车辆状态模型。
- 能为车辆配置成都市监管编号和 TLS 证书。
- 能与成都市监管平台建立 TCP/TLS 连接并发送准静态参数。
- 能按策略持续发送运行状态和故障消息。
- 能在后台查看每辆车源数据时间、标准状态时间、最近上报时间、连接状态和错误原因。
- 能查询原始数据、标准化数据、上报日志和告警。
- 新增厂商或城市监管平台时，不需要改动已有适配器核心逻辑，只需新增适配器和映射配置。
